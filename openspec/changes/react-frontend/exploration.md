# Exploration: React + Vite Frontend for AlphaRadar

## Current State

AlphaRadar is a Python FastAPI stock analysis MVP with a hexagonal architecture (domain → application → infrastructure). The backend exposes a REST API under `/api/` with the following endpoints:

### API Surface Summary

| Endpoint | Method | Request | Response | Description |
|----------|--------|---------|----------|-------------|
| `/api/health/live` | GET | — | `{status: "alive", version: "0.1.0"}` | Liveness check |
| `/api/stocks/{symbol}` | GET | path: `symbol` (string) | `StockResponse {symbol, name, exchange}` | Stock metadata |
| `/api/recommendations/{symbol}` | GET | path: `symbol` (string) | `RecommendationResponse {symbol, action, score, confidence, reasoning, scores: {technical, fundamental, sentiment, final_score}}` | Buy/sell recommendation |
| `/api/portfolio` | GET | — | `{portfolio: {name, positions: [StoredPosition], total_value, total_pnl, storage}}` | View portfolio |
| `/api/portfolio/positions` | POST | body: `PositionCreate {symbol, quantity, avg_buy_price}` | `PositionResponse {id, symbol, quantity, avg_buy_price, current_price}` | Add position |
| `/api/portfolio/positions/{position_id}` | DELETE | path: `position_id` (int) | 204 No Content | Delete position |

**Key data models:**
- `Stock`: symbol, name, exchange, info (optional StockInfo), history (list of OHLCV)
- `StockInfo`: pe_ratio, eps, market_cap, revenue, debt_to_equity, dividend_yield, sector, industry
- `OHLCV`: timestamp, open, high, low, close, volume
- `Recommendation`: symbol, action (BUY/SELL/HOLD), score, confidence, reasoning, scores, created_at, is_active
- `AnalysisScores`: technical (0-100), fundamental (0-100), sentiment (-1 to +1), final_score
- `Position`: symbol, quantity, avg_buy_price, current_price (computed unrealized_pnl, market_value)
- `StoredPosition`: id, symbol, quantity, avg_buy_price, current_price

**Current frontend:** Swagger UI (auto-generated by FastAPI) at `/docs`. No custom frontend exists.

**Infrastructure:** Docker Compose orchestrates app (FastAPI), PostgreSQL, Redis, Celery worker, and beat scheduler. The app is served via uvicorn on port 8000.

## Affected Areas

Adding a React + Vite frontend would involve:

- **New directory:** `frontend/` at project root (React + Vite app)
- **Modified:** `src/alpharadar/api/main.py` — mount static files for production serving
- **Modified:** `Dockerfile` — multi-stage build to include frontend build
- **Modified:** `docker-compose.yml` — optional: add frontend dev service or adjust app service
- **New:** `frontend/package.json`, `frontend/vite.config.ts`, `frontend/src/`, etc.
- **New:** `tests/e2e/` — Playwright test suite
- **Possibly modified:** `.gitignore` — add `node_modules/`, `dist/`

## Approaches

### 1. Frontend Stack: React + Vite (User-chosen)

| Aspect | Choice | Tradeoffs |
|--------|--------|-----------|
| **Framework** | React 19 + Vite | Fast HMR, modern tooling, large ecosystem. Steeper learning curve than vanilla JS. |
| **Styling** | Tailwind CSS | Rapid prototyping, consistent design, larger bundle (mitigated by purging). vs CSS Modules (scoped, smaller bundle, more CSS writing). vs styled-components (dynamic styles, runtime overhead). |
| **State Management** | Zustand | Minimal boilerplate, TypeScript-first, small bundle (~1KB). vs React Context (built-in, but causes re-renders). vs Redux (powerful, heavy for this MVP). |
| **Data Fetching** | TanStack Query (React Query) | Caching, background refetch, optimistic updates. vs manual useEffect (simpler but no caching). vs SWR (similar to React Query). |
| **Routing** | React Router v7 | Client-side routing, lazy loading. vs TanStack Router (type-safe, newer). |
| **Forms** | React Hook Form + Zod | Performant validation, integrates with Pydantic schemas. vs Formik (heavier). |

### 2. Integration Approach

| Approach | Pros | Cons | Effort |
|----------|------|------|--------|
| **Vite proxy (dev) + FastAPI static mount (prod)** | Simple, no CORS issues, single origin in prod | Requires build step for prod | Low |
| **CORS + separate origins** | Decoupled deployments | CORS config, cookie issues, harder to deploy | Medium |
| **BFF (Backend-for-Frontend) pattern** | API aggregation, tailored responses | Over-engineering for MVP | High |

### 3. E2E Testing

| Tool | Pros | Cons |
|------|------|------|
| **Playwright** | Multi-browser, auto-wait, network interception, excellent DX | Heavier than Cypress, learning curve |
| **Cypress** | Simpler API, time-travel debugging | Single-tab limitation, slower, paid features |

## Proposed UI Pages and Data Needs

### Page 1: Dashboard (`/`)
- **Purpose:** Portfolio overview at a glance
- **API calls:** `GET /api/portfolio`
- **Data displayed:** Total value, total P&L, list of positions with symbol, quantity, current price, unrealized P&L
- **Actions:** Navigate to stock detail, add position

### Page 2: Stock Search & Detail (`/stocks/:symbol`)
- **Purpose:** Look up a stock and get recommendation
- **API calls:** `GET /api/stocks/{symbol}`, `GET /api/recommendations/{symbol}`
- **Data displayed:** Stock name, exchange, recommendation action (BUY/SELL/HOLD), score, confidence, reasoning, analysis scores breakdown
- **Actions:** Add to portfolio, go back to search

### Page 3: Portfolio Management (`/portfolio`)
- **Purpose:** Manage positions
- **API calls:** `GET /api/portfolio`, `POST /api/portfolio/positions`, `DELETE /api/portfolio/positions/{id}`
- **Data displayed:** Full position list with edit/delete actions, add position form
- **Actions:** Add, edit (future), delete positions

### Page 4: Health Check (`/health`) — Optional
- **Purpose:** System status (could be admin-only)
- **API calls:** `GET /api/health/live`
- **Data displayed:** API status, version

### State Management Approach
- **Server state:** TanStack Query for API data (caching, refetching, loading states)
- **Client state:** Zustand for UI state (selected symbol, form state, theme)
- **No global state needed** for this MVP — most data is server-derived

| Layer | Choice | Rationale |

| Build tool | Vite | Fast HMR, native ESM, excellent DX |

## Integration Approach

### Development
- Vite dev server on port 5173
- Proxy `/api` to FastAPI on port 8000 via `vite.config.ts`:
  ```ts
  server: {
    proxy: {
      '/api': 'http://localhost:8000'
    }
  }
  ```
- No CORS issues (same origin via proxy)

### Production
- Build frontend: `cd frontend && npm run build` → `frontend/dist/`
- FastAPI serves static files:
  ```python
  from fastapi.staticfiles import StaticFiles
  app.mount("/", StaticFiles(directory="frontend/dist", html=True), name="static")
  ```
- Single container, single origin, no proxy needed
- Dockerfile multi-stage: Node build stage → Python runtime stage

### E2E Testing with Playwright
- `tests/e2e/` directory with Playwright config
- Tests run against running app (Docker Compose or local dev)
- Playwright config points to `http://localhost:5173` (dev) or `http://localhost:8000` (prod-like)
- Test scenarios: stock search, recommendation display, portfolio CRUD, error handling

## Risks

1. **API schema changes:** Backend is pre-MVP; endpoints may change. Mitigation: Generate TypeScript types from OpenAPI schema.
2. **CORS in development:** If proxy misconfigured. Mitigation: Vite proxy handles this automatically.
3. **Bundle size:** Tailwind + React could be large. Mitigation: Tree-shaking, code splitting, lazy loading.
4. **Single-user constraint:** Current portfolio is process-local (in-memory). Frontend must handle empty state gracefully.
5. **No authentication:** MVP is single-user, loopback-only. Frontend should not assume auth but must handle 401/403 if added later.
6. **Mobile responsiveness:** Not specified. Tailwind helps, but needs deliberate responsive design.
7. **Error handling:** Backend returns specific error codes (404, 503). Frontend must map these to user-friendly messages.

## Unknowns

1. **Design system:** No existing UI/UX design. Need to decide on color palette, typography, spacing.
2. **Charting requirements:** Stock price charts? Recommendation score visualization? May need Recharts or similar.
3. **Offline support:** Not required for MVP but could be a future consideration.
4. **Deployment target:** Same Docker Compose? Separate hosting? Affects build and serving strategy.

## Recommendation

Proceed with **React + Vite + Tailwind CSS + Zustand + TanStack Query**. This stack provides:
- Fast development velocity (Vite HMR, Tailwind utility classes)
- Excellent TypeScript support (Zod schemas, TanStack Query types)
- Minimal bundle overhead (Zustand ~1KB, Tailwind purged)
- Simple integration (Vite proxy dev, FastAPI static mount prod)
- Reliable E2E testing (Playwright)

The approach is low-effort and aligns with the MVP scope. Start with the Dashboard and Stock Detail pages, then add Portfolio Management.
